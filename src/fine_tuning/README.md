# Fine-tuning

Reinforcement-learning fine-tuning for the Talk2PowerSystem agent. This pipeline trains an open-weight LLM (e.g. `Qwen/Qwen2.5-14B-Instruct`) to drive the LangGraph ReAct agent defined in [agent_hf.py](agent_hf.py) against a GraphDB-backed CIM knowledge graph, rewarding trajectories whose SPARQL tool outputs match the reference results.

Training is implemented with [OpenPipe ART](https://github.com/OpenPipe/ART) (GRPO-style rollouts) on top of Unsloth + vLLM, and answer quality is judged by an LLM-as-reward model.

## Setup

```bash
# Python 3.12
git clone https://github.com/statnett/Talk2PowerSystem_LLM.git
cd Talk2PowerSystem_LLM
git reset --hard 9fe38f3c4c8f01c8410bb09ddc9120b29453e4bf
poetry install
pip install -r src/fine_tuning/requirements.txt
```

A CUDA GPU is required (Unsloth + vLLM). The default config assumes a single A40 (see `--gpus-per-node=1` in [run_train.sh](run_train.sh)).

### External services

* **GraphDB** reachable at the URL set under `graphdb.base_url` in [dev_open_models_no_ontology.yaml](dev_open_models_no_ontology.yaml) (default `http://localhost:7200`, repository `cim`), loaded with the CIM dataset and the autocomplete / RDF-rank plugins enabled.
* **vLLM / HF-compatible inference endpoint** at `llm.hugging_face_endpoint` (default `http://localhost:8000/v1/`). ART also spins up its own vLLM engine for the trainable model; the endpoint in the YAML is used only by the agent wiring.

## Data

### `train_rl.json` — RL dataset

A flat JSON array. Each record carries one question together with its gold SPARQL step and the canonical results:

```jsonc
{
  "question_text":      "What are the disconnectors linked to the topological node 734aa242?",
  "sparql_step":        "SELECT DISTINCT ?disconnector WHERE { ... }",
  "sparql_step_output": "{\"head\": {...}, \"results\": {\"bindings\": [...] }}",
  "required_columns":   ["disconnector"]
}
```

`train.py` reads it with `datasets.Dataset.from_json` and the rollout uses `sparql_step_output` + `required_columns` as the reference the agent's SPARQL tool call must match.

### `question_instances_full_M5_202602.yaml` — KGQA benchmark

Generated by the KGQA builder. Each template lists paraphrases, reference steps, and the ontology subset needed to answer the question. Not consumed directly by `train.py` — it is the source used to derive `train_rl.json` and `q2ontology.json`.

### `q2ontology.json` — per-question ontology subset

Generated by the KGQA builder. Maps the exact `question_text` to a small Turtle fragment of the ontology (only the classes/properties required for that question). Passed to `train.py` via `--subsets q2ontology.json`; at rollout time it is substituted into the `{ontology_schema}` placeholder in the assistant prompt so the model sees only the relevant slice of the schema instead of the full ontology.

## Training

### Launching

```bash
bash run_train.sh
```

which runs:

```bash
python train.py \
  --model_name "Qwen/Qwen2.5-14B-Instruct" \
  --name      "Qwen2.5-14Blr_5e-7_gspo_kl_0.01_subset_speed" \
  --subsets   "q2ontology.json"
```

All CLI flags are `fire`-wired directly to `main(...)` in [train.py:227](train.py#L227). The most relevant ones:

| Flag                                      | Default                            | Meaning                                                               |
| ----------------------------------------- | ---------------------------------- | --------------------------------------------------------------------- |
| `--model_name`                            | *(required)*                       | HF id / local path of the base model.                                 |
| `--name`                                  | `None`                             | ART run name (checkpoints land under `./.art/statnett/<name>/`).      |
| `--train`                                 | `train_rl.json`                    | Training set path.                                                    |
| `--subsets`                               | `None`                             | Optional `question -> ontology-subset` JSON (pass `q2ontology.json`). |
| `--path_to_yaml_config`                   | `dev_open_models_no_ontology.yaml` | Agent runtime config.                                                 |
| `--max_tokens`                            | `20000`                            | Max sequence length for Unsloth + vLLM.                               |
| `--gpu_memory_utilization`                | `0.70`                             | vLLM GPU memory fraction.                                             |
| `--r`, `--lora_alpha`, `--target_modules` | `16`, `16`, all attn+MLP           | LoRA adapter shape.                                                   |
| `--learning_rate`                         | `5e-7`                             | GRPO learning rate.                                                   |
| `--beta`                                  | `0.01`                             | KL coefficient.                                                       |
| `--rollouts_per_group`                    | `8`                                | Rollouts generated per training item.                                 |
| `--groups_per_step`                       | `1`                                | Dataset items per optimiser step.                                     |
| `--num_epochs`                            | `1`                                | Passes over the dataset.                                              |
| `--max_time`                              | `200`                              | Seconds; rollouts exceeding this get `reward = 0`.                    |

### What a rollout does

`rollout(...)` in [train.py:99](train.py#L99):

1. Builds a `Talk2PowerSystemAgent` with a per-rollout `MemorySaver`.
2. If `q2subset` is provided, replaces `{ontology_schema}` in the system prompt with the question-specific Turtle subset; otherwise the placeholder is left intact.
3. Invokes the LangGraph agent with a recursion limit of 25.
4. Scans the resulting messages for `ToolMessage`s, parses each as SPARQL JSON results, and computes the best `compare_sparql_results(...)` score against `sparql_step_output` (from `graphrag_eval`).
5. If any tool call produced a positive score, asks the base model (via the vLLM OpenAI-compatible endpoint) to judge the final answer against `RESPONSE_EVAL_PROMPT` — primarily a **language-purity / formatting** filter (English-only, no stray CJK/Cyrillic, no raw special tokens). A failed check zeros the reward.
6. Scales the reward by wall-clock time (`reward /= total_time`) and zeroes it when `total_time > max_time`, so the policy is pushed toward both correctness *and* speed.
7. Writes the full trajectory to `full_trace.jsonl` for inspection.

Streaming averages for raw reward, time-scaled reward, and elapsed time are logged to stdout after every rollout.

### Outputs

* `./.art/statnett/<name>/` — ART project dir with LoRA checkpoints and step metadata.
* `full_trace.jsonl` — last rollout's message trace.
* Weave traces under project `statnett` (via `@weave.op()` on `rollout`).

## Merging the adapter

Once training is done, produce a standalone checkpoint with [merge.py](merge.py):

```bash
python merge.py \
  --adapter_path ./.art/statnett/<name>/checkpoints/<step> \
  --output_path  ./merged/<name> \
  --max_seq_length 20000 \
  --method merged_16bit          # or merged_4bit
```

`merge.py` uses Unsloth's `FastLanguageModel.save_pretrained_merged` — the adapter config in `--adapter_path` pins the base model, so no `--model_name` flag is needed.

## Agent configuration ([dev_open_models_no_ontology.yaml](dev_open_models_no_ontology.yaml))

Top-level sections:

* `graphdb` — endpoint, repository, timeouts, optional basic auth. Populated into `GraphDBSettings` in [agent_hf.py:31](agent_hf.py#L31).
* `tools.ontology_schema.file_path` — Turtle file loaded by `OntologySchemaAndVocabularyTool`. The "no_ontology" variant points at `ontology/empty.ttl`; the per-question subset is injected through the prompt instead.
* `tools.autocomplete_search` — property path + SPARQL template for the autocomplete plugin.
* `tools.retrieval_search`, `tools.cognite` — optional extra tools (disabled by default; Cognite wiring is currently commented out in the agent).
* `llm.*` — used only for the Azure/OpenAI client path; during training ART manages the trainable model directly and the agent pulls logits from ART's own vLLM engine via `art.langgraph.init_chat_model(model.name, ...)` (see [agent_hf.py:266](agent_hf.py#L266)).
* `prompts.assistant_instructions` — the system prompt, including the `{ontology_schema}` placeholder that `train.py` fills in per question.

| --------------------------------- | ------------------------- | ------------------------------------------------------------------------------------- |
| `UNSLOTH_VLLM_STANDBY=1`          | [train.py:4](train.py#L4) | Keeps Unsloth and vLLM co-resident so rollouts and training share one process.        |

## Troubleshooting

* **`Error No ontology subset for question:` in logs** — a question in `train_rl.json` is missing from `q2ontology.json`. Regenerate `q2ontology.json` from the KGQA builder against the same question set.
* **Rewards stuck at 0** — verify GraphDB is reachable and the `cim` repository is loaded; `SparqlQueryTool` errors are swallowed inside `rollout` and silently produce zero reward.
